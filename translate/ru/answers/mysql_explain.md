# EXPLAIN в MySQL

Команда `EXPLAIN` предоставляет информацию о том, как MySQL выполняет запрос. Она работает с операторами `SELECT`, `DELETE`, `INSERT`, `REPLACE` и `UPDATE`. Понимание её вывода — ключ к оптимизации базы данных.

## Основные столбцы вывода EXPLAIN

| Столбец | Описание |
| :--- | :--- |
| **id** | Порядковый номер каждого `SELECT` внутри запроса. Помогает отследить порядок выполнения. |
| **select_type** | Тип запроса (например, SIMPLE, PRIMARY, SUBQUERY, DERIVED, UNION). |
| **table** | Название таблицы, к которой относится строка. Может быть производной таблицей (например, `<derivedN>`). |
| **partitions** | Партиции, из которых будут выбираться данные. |
| **type** | Тип соединения (как MySQL выбирает данные). **Критически важен для производительности.** |
| **possible_keys** | Индексы, которые MySQL *мог бы* использовать для поиска строк. |
| **key** | Индекс, который MySQL *фактически* решил использовать. |
| **key_len** | Длина выбранного ключа в байтах. Позволяет проверить, все ли части составного индекса используются. |
| **ref** | Столбцы или константы, которые сравниваются с индексом, указанным в `key`. |
| **rows** | Оценочное количество строк, которые MySQL должен просмотреть. |
| **filtered** | Оценочный процент строк, отфильтрованных условием таблицы. |
| **Extra** | Дополнительная информация о том, как MySQL разрешает запрос. |

## Подробное описание столбцов

### 1. id и select_type
- **SIMPLE**: Простой запрос без подзапросов или объединений.
- **PRIMARY**: Самый внешний `SELECT` в сложном запросе.
- **SUBQUERY**: `SELECT` в подзапросе (не в блоке `FROM`).
- **DERIVED**: Подзапрос в блоке `FROM` (производная таблица).
- **UNION**: Второй или последующий `SELECT` в объединении `UNION`.
- **DEPENDENT SUBQUERY**: Подзапрос, результат которого зависит от внешнего запроса (часто работает медленно).
- **UNCACHEABLE SUBQUERY**: Подзапрос, результат которого не может быть закеширован (например, при использовании `RAND()`).

### 2. Типы соединений (столбец `type`)
Отсортированы от **лучших к худшим** по производительности:
1.  **system**: В таблице только одна строка (частный случай `const`).
2.  **const**: В таблице не более одной совпадающей строки (например, поиск по первичному ключу или уникальному индексу с константами).
3.  **eq_ref**: Для каждой комбинации строк из предыдущих таблиц читается одна строка из этой таблицы (используется уникальный индекс).
4.  **ref**: Читаются все строки с совпадающими значениями индекса (используется неуникальный индекс или префикс).
5.  **fulltext**: Соединение выполняется с использованием индекса `FULLTEXT`.
6.  **ref_or_null**: Похоже на `ref`, но дополнительно ищет строки с `NULL`.
7.  **index_merge**: Используется несколько индексов, и их результаты объединяются.
8.  **unique_subquery**: Для подзапросов `IN`, возвращающих уникальные результаты.
9.  **range**: Извлекаются только строки в заданном диапазоне с использованием индекса (например, при использовании `>`, `<`, `BETWEEN`, `IN`).
10. **index**: Полное сканирование индекса (лучше, чем `ALL`, так как индекс обычно меньше таблицы).
11. **ALL**: Полное сканирование таблицы (самый медленный вариант).

### 3. Важные значения в столбце "Extra"
- **Using index**: Запрос разрешается только с использованием индекса (покрывающий индекс). Обращение к самой таблице не требуется.
- **Using where**: Используется условие `WHERE` для фильтрации строк *после* их получения из движка хранения.
- **Using temporary**: MySQL необходимо создать внутреннюю временную таблицу для хранения результата (часто при `GROUP BY` или `ORDER BY` по неиндексированным столбцам). **Тревожный сигнал!**
- **Using filesort**: MySQL должен выполнить дополнительный проход, чтобы найти способ извлечь строки в отсортированном порядке. **Тревожный сигнал!**
- **Select tables optimized away**: Оптимизатор определил, что доступ к таблицам не требуется (например, `MIN()` по индексированному столбцу).
- **FirstMatch(table_name)**: Оптимизация semi-join, которая прекращает поиск, как только найдено первое совпадение в подзапросе.

## Современные возможности MySQL (8.0+)

### EXPLAIN FORMAT=TREE
Выводит план выполнения в виде иерархического дерева. Показывает стоимость каждой операции и оценочное количество строк.
```sql
EXPLAIN FORMAT=TREE SELECT ...
```

### EXPLAIN ANALYZE
Самый мощный инструмент профилирования. Он **фактически выполняет** запрос и выводит:
- **Actual time**: фактическое время получения первой строки и всех строк.
- **Actual rows**: фактическое количество возвращённых строк.
- **Loops**: количество циклов выполнения операции.
```sql
EXPLAIN ANALYZE SELECT ...
```

## Как читать EXPLAIN
1.  **Читайте сверху вниз**: MySQL выполняет план в указанном порядке.
2.  **Проверяйте `type`**: Стремитесь к `const`, `eq_ref` или `ref`. Если видите `ALL` или `index`, подумайте о добавлении индексов.
3.  **Смотрите на `rows` и `filtered`**: Перемножьте их, чтобы понять, сколько строк передаётся на следующий этап.
4.  **Следите за предупреждениями в `Extra`**: `Using temporary` и `Using filesort` обычно означают, что нужны лучшие индексы для `GROUP BY` или `ORDER BY`.
5.  **Используйте `SHOW WARNINGS`**: После `EXPLAIN` выполните `SHOW WARNINGS`, чтобы увидеть, как MySQL переписал ваш запрос.
