# SQL Представления и Материализованные представления

SQL Представления (Views) и Материализованные представления (Materialized Views) — это объекты базы данных, используемые для упрощения доступа к данным, повышения безопасности и оптимизации производительности.

---

## 1. Обычные SQL Представления (Виртуальные таблицы)

**Представление (View)** — это виртуальная таблица, основанная на результатах SQL-запроса. Оно не хранит данные физически; вместо этого оно сохраняет определение запроса и выполняет его каждый раз, когда вы обращаетесь к представлению.

### Назначение и важность
- **Упрощение:** Скрывает сложные Join-соединения и вычисления от конечного пользователя.
- **Безопасность:** Ограничивает доступ к определенным строкам или колонкам (например, скрытие колонки `salary`).
- **Консистентность и Абстракция:** Обеспечивает единый интерфейс, даже если структура базовых таблиц меняется. Это критично для **устаревших систем (legacy)**, где вы можете переименовать колонки или таблицы, не ломая само приложение.
- **Целостность данных:** Использование `WITH CHECK OPTION` гарантирует, что измененные через представление данные останутся в рамках условий этого представления.
- **Централизация логики:** Позволяет следовать принципу DRY (Don't Repeat Yourself), собирая сложную бизнес-логику в одном месте.

### Как использовать Представления

#### Создание Представления
```sql
-- Простое представление (из одной таблицы)
CREATE VIEW active_users AS
SELECT id, username, email
FROM users
WHERE status = 'active';

-- Сложное представление (из нескольких таблиц)
CREATE VIEW order_details AS
SELECT o.id, u.username, p.product_name, o.amount
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id;
```

#### Управление Представлениями
```sql
-- Список всех представлений (пример для MySQL)
SHOW FULL TABLES WHERE table_type LIKE '%VIEW';

-- Получение списка представлений через information_schema (Стандарт SQL)
SELECT table_name 
FROM information_schema.views 
WHERE table_schema = 'имя_вашей_бд';

-- Удаление представления
DROP VIEW active_users;

-- Обновление определения представления
CREATE OR REPLACE VIEW active_users AS
SELECT id, username, email, last_login
FROM users
WHERE status = 'active';
```

### Обновление данных через Представления
Не все представления позволяют изменять данные. Чтобы представление было обновляемым, оно должно соответствовать нескольким правилам:
1. Запрос `SELECT` не должен содержать `GROUP BY`, `DISTINCT` или агрегатные функции (`SUM`, `COUNT`).
2. Представление должно быть основано на **одной** базовой таблице (в большинстве СУБД).
3. Если вы хотите делать `INSERT`, представление должно включать все колонки базовой таблицы, помеченные как `NOT NULL`.

**`WITH CHECK OPTION`:**
Этот пункт предотвращает выполнение `UPDATE` или `INSERT`, после которых строка перестала бы попадать под условия фильтрации представления.
```sql
CREATE VIEW high_salary_employees AS
SELECT * FROM employees WHERE salary > 5000
WITH CHECK OPTION;

-- Это действие завершится ОШИБКОЙ, так как оно нарушает условие WHERE представления
UPDATE high_salary_employees SET salary = 4000 WHERE id = 1;
```

### Плюсы и минусы
- **Плюсы:** Всегда актуальные данные, отсутствие затрат на хранение, повышенная безопасность.
- **Минусы:** Падение производительности при сложных запросах (базовый запрос выполняется каждый раз).

## 2. Материализованные представления (Кешированные таблицы)

**Материализованное представление (Materialized View)** — это объект базы данных, который содержит *фактические результаты* запроса, сохраненные на диске как физическая таблица.

### Ключевые различия:
| Функция | Обычное представление | Материализованное представление |
| :--- | :--- | :--- |
| **Хранение** | Виртуальное (только определение запроса) | Физическое (кешированные данные) |
| **Производительность** | Зависит от сложности запроса | Чрезвычайно высокая (как у таблицы) |
| **Актуальность данных** | В реальном времени (всегда актуально) | Периодическая (обновляется при обновлении) |
| **Индексы** | Ограничено (использует индексы базовых таблиц) | Поддерживает собственные индексы на любой колонке |

### Как использовать материализованные представления?

#### Синтаксис PostgreSQL (9.3+)
```sql
-- 1. Создание материализованного представления
CREATE MATERIALIZED VIEW sales_summary AS
SELECT product_id, SUM(quantity) as total_sold
FROM sales
GROUP BY product_id;

-- 2. Запрос к нему как к обычной таблице
SELECT * FROM sales_summary WHERE total_sold > 1000;

-- 3. Обновление данных (вручную)
REFRESH MATERIALIZED VIEW sales_summary;

-- 4. Конкурентное обновление (PostgreSQL 9.4+, не блокирует представление на чтение)
CREATE UNIQUE INDEX idx_product_id ON sales_summary(product_id);
REFRESH MATERIALIZED VIEW CONCURRENTLY sales_summary;
```

#### Синтаксис Oracle
```sql
CREATE MATERIALIZED VIEW mv_sales
REFRESH FAST START WITH SYSDATE
NEXT SYSDATE + 1 -- Автоматическое обновление каждый день
AS SELECT * FROM sales;
```

#### Обходной путь в MySQL
MySQL не поддерживает материализованные представления нативно. Однако их можно имитировать следующим образом:
1. **Сводные таблицы:** Обычная таблица, наполняемая через `INSERT INTO ... SELECT`.
2. **Триггеры:** Для обновления сводной таблицы при изменении базовых таблиц.
3. **Планировщик:** Обновление таблицы по расписанию (Cron или Event Scheduler).

### Плюсы и минусы
- **Плюсы:** Значительный прирост производительности для тяжелых агрегаций, снижение нагрузки на базовые таблицы.
- **Минусы:** Неактуальность данных (stale), затраты на хранение, сложность обслуживания.

## FAQ на основе статей

### Когда использовать обычное View, а когда Материализованное?
Используйте **обычное View** для безопасности (скрытие колонок) или упрощения небольших/средних запросов, где критически важна актуальность данных. Используйте **Материализованное представление** для сложной аналитики, отчетов или агрегаций на больших наборах данных, где производительность важнее 100% свежести данных.

### Можно ли вставить данные в View, объединяющее несколько таблиц?
Обычно нет. Большинство баз данных разрешают обновления только для «простых» представлений, которые отображаются на одну базовую таблицу.

### Быстрее ли View, чем обычный запрос?
Нет. View — это и есть обычный запрос. Однако **Материализованное представление** намного быстрее, так как оно не пересчитывает результат каждый раз.
