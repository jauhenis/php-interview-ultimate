# SQL Literacy: Блокировки, Изоляция и Идемпотентность

Глубокое понимание работы баз данных необходимо для проектирования отказоустойчивых и масштабируемых систем. Неправильная обработка параллельных запросов приводит к потере данных, превышению лимитов и нестабильной работе API.

---

## 1. Зачем это нужно Senior-разработчику?

Знание «внутренностей» SQL (MVCC, ID транзакций, механизмы блокировок) позволяет:
- Избегать трудновоспроизводимых багов (Race Conditions).
- Оптимизировать нагрузку на БД, выбирая минимально необходимые блокировки.
- Строить надежные интеграции с внешними системами (очереди, платежные шлюзы).

---

## 2. Сценарий №1: Race Condition при обновлении (Lost Update)

### Описание задачи
Система бронирования. Пользователь нажимает «Занять слот». Приложение проверяет, свободен ли он (`speaker_id IS NULL`), и записывает ID пользователя.

**Проблема (Race Condition):**
При одновременном запросе два процесса могут прочитать `NULL`. Оба решат, что слот свободен, и выполнят `UPDATE`. В итоге данные первого пользователя будут перезаписаны данными второго.

### Решения:
1. **Pessimistic Locking**: `SELECT ... FOR UPDATE`. Блокирует строку до конца транзакции. Другие процессы будут ждать.
2. **Atomic Update (Рекомендуется)**:
   ```sql
   UPDATE meetup_slots 
   SET speaker_id = :user_id 
   WHERE id = :id AND speaker_id IS NULL;
   ```
3. **Проверка Rows Affected**: Если обновлено 0 строк — значит слот уже занят другим. Пользователю нужно вернуть ошибку или предложить другой вариант.

---

## 3. Сценарий №2: Идемпотентность и повторные запросы

### Описание задачи
Сеть нестабильна. Клиент может отправить запрос дважды из-за таймаута, или сообщение в очереди может быть доставлено повторно (at-least-once).

### Решение:
1. **Unique Index**: Создайте уникальный индекс на бизнес-ключи (например, `meetup_id` + `user_id`).
2. **Обработка Duplicate Key**:
   - При вставке ловим ошибку дубликата.
   - Если данные те же самые — возвращаем `200 OK`. Цель достигнута, повторная вставка не нужна.
   - Это делает API **идемпотентным**.

## 4. Сценарий №3: Лимиты и конкурентный подсчет

### Описание задачи
Ограничение количества мест на событии (например, 100). Нужно гарантировать, что 101-й пользователь не сможет записаться.

**Проблема:**
`SELECT COUNT(*)` внутри транзакции не видит данные других параллельных транзакций (зависит от уровня изоляции). Два процесса могут увидеть «99 мест занято» и оба вставят записи.

### Решения:
1. **Блокировка родителя**: `SELECT * FROM events WHERE id = :id FOR UPDATE`. Все регистрации на это событие выстраиваются в очередь.
2. **Денормализация + CHECK**:
   - Добавляем поле `booked_count` и `CHECK (booked_seats <= max_seats)`.
   - Обновляем счетчик при каждой записи:
     ```sql
     UPDATE events SET booked_count = booked_count + 1 WHERE id = :id;
     ```
   - База сама откатит транзакцию при нарушении лимита.

## 5. Глубокое погружение: MVCC и скрытые колонки

Для работы конкурентности (одна транзакция читает, другая пишет одновременно) базы данных используют **MVCC (Multi-Version Concurrency Control)**.

### PostgreSQL
Хранит все версии строк в самой таблице. Скрытые колонки:
- **`xmin`**: ID транзакции, которая создала версию строки.
- **`xmax`**: ID транзакции, которая удалила или обновила строку.
- **`ctid`**: физический адрес строки на диске.

### MySQL (InnoDB)
Хранит в таблице только актуальную версию, а старые выносит в **Undo Log**:
- **`DB_TRX_ID`**: ID последней транзакции, создавшей версию.
- **`DB_ROLL_PTR`**: указатель на предыдущую версию в Undo Log (используется для восстановления состояния).
- **`DB_ROW_ID`**: скрытый ID строки (если нет первичного ключа).

## 6. Распределенные блокировки: SQL vs Redis

### Когда использовать Redis?
- **Rate Limiting**: Ограничение частоты запросов, чтобы не «положить» БД.
- **Circuit Breaker**: Временная блокировка доступа к ресурсу при сбоях.
- **Синхронизация Cron**: Чтобы одна задача не запустилась на нескольких серверах одновременно.

### Почему НЕ использовать Redis для консистентности данных?
1. **Нет атомарности**: Нельзя одной транзакцией заблокировать ключ в Redis и обновить строку в SQL.
2. **Риски TTL**: Блокировка в Redis может истечь по таймауту раньше, чем завершится долгая транзакция в БД.
3. **Лишняя сложность**: БД уже умеет обеспечивать консистентность (Unique Index, Check, Row Locks).

## 7. Чек-лист для разработки

- [ ] **CHECK и Unique Index**: используйте для защиты данных.
    - [ ] Избегайте лишних автоинкрементных ID (autogen ID), если они не нужны.
    - [ ] Используйте составные первичные ключи (Composite PK).
    - [ ] Добавляйте `operation_id`, `campaign_id` или `idempotency_key`.
- [ ] **Минимизация FOR UPDATE**: обходитесь без него, если это возможно.
    - [ ] Это экономит ресурсы на ожидание и удержание блокировок.
    - [ ] Обычные `SELECT` можно выполнять на реплике.
- [ ] **Rows Affected**: всегда проверяйте после `UPDATE`. Помните, что `0` — это иногда валидный результат.
- [ ] **Ошибки SQL**: обрабатывайте нарушения констрейнтов (Constraint Violation), ошибки сериализации (Serialization Error) и Deadlocks.

---

## 8. Чек-лист для System Design

1.  **Конкурентность**: есть ли одновременный доступ к одной сущности от разных пользователей?
2.  **Фрод**: может ли пользователь попытаться «обмануть» систему (например, двойное списание)?
3.  **Уровень консистентности**: выбор между **Strong Consistency** и **Eventual Consistency**.
4.  **Идемпотентность**: закладывайте её на уровне архитектуры.
5.  **Варианты решений**: продумайте несколько альтернативных подходов.
6.  **Параллельная проверка**: обязательно проверяйте работу транзакций в параллельном режиме.
7.  **Режимы отказа**: что произойдет, если связь оборвется между коммитом в БД и ответом клиенту?
